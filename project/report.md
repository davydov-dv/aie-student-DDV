## 2. Постановка задачи и контекст

### 2.1 Предметная область и задача

**Задача:** детекция сетевых аномалий и атак в реальном времени на основе статистических метрик сетевого трафика.

**Пользователь сервиса:**
- Сетевой администратор, получающий алерты о подозрительной активности.
- SIEM-система (Security Information and Event Management), интегрирующая предсказания в общий контур безопасности.
- SOC-аналитик, использующий API для расследования инцидентов.

**Сценарий использования:** сервис принимает на вход метрики сетевого потока (длительность, количество байт/пакетов, протокол, состояние соединения и др.) и возвращает бинарное предсказание с указанием уверенности модели.

### 2.2 Формулировка задачи в терминах ML

**Входные данные:** 42 признака сетевого потока (flow), описывающих соединение:
- Числовые (39): `dur`, `spkts`, `dpkts`, `sbytes`, `dbytes`, `rate`, `sttl`, `dttl`, `sload`, `dload`, `sloss`, `dloss`, `sinpkt`, `dinpkt`, `sjit`, `djit`, `swin`, `stcpb`, `dtcpb`, `dwin`, `tcprtt`, `synack`, `ackdat`, `smean`, `dmean`, `trans_depth`, `response_body_len`, `ct_srv_src`, `ct_state_ttl`, `ct_dst_ltm`, `ct_src_dport_ltm`, `ct_dst_sport_ltm`, `ct_dst_src_ltm`, `is_ftp_login`, `ct_ftp_cmd`, `ct_flw_http_mthd`, `ct_src_ltm`, `ct_srv_dst`, `is_sm_ips_ports`.
- Категориальные (3): `proto`, `service`, `state`.

**Целевая переменная:** бинарная метка `label ∈ {0, 1}`, где 0 — нормальный трафик, 1 — атака.

**Ограничения:**
- **Скорость инференса:** не более 50 мс на один запрос (требование real-time детекции).
- **Интерпретируемость:** возможность объяснить предсказание через важность признаков.
- **Точность:** минимизация числа пропущенных атак (высокий Recall) при приемлемом уровне ложных срабатываний.

### 2.3 Целевые метрики качества

| Метрика | Зачем нужна |
|---------|-------------|
| **F1-Score** (основная) | Баланс Precision и Recall при несбалансированных классах (~44% атак, ~56% нормы) |
| **Recall** | Доля обнаруженных атак — критична для безопасности (пропуск атаки дороже ложного алерта) |
| **Precision** | Доля корректных предсказаний атак — важна для снижения ложных алертов |
| **Accuracy** | Вспомогательная метрика для общего понимания качества |

ROC-AUC не используется как основная, так как в продакшне модель работает с фиксированным порогом, а не ранжирует примеры.

### 3.1 Источник данных

Используется открытый академический датасет **UNSW-NB15** [1], созданный Australian Centre for Cyber Security (UNSW Canberra). Трафик генерировался инструментом IXIA PerfectStorm, перехватывался через Tcpdump (100 ГБ PCAP) и преобразовывался в признаки инструментами Argus и Bro-IDS.

- **Ссылка:** https://www.kaggle.com/datasets/mrwellsdavid/unsw-nb15
- **Лицензия:** свободное использование в академических целях
- **Расположение в проекте:** `data/raw/UNSW_NB15/`

### 3.2 Структура данных

| Файл | Размер | Назначение |
|------|--------|------------|
| `UNSW_NB15_training-set.csv` | 175 341 записей | Обучающая выборка |
| `UNSW_NB15_testing-set.csv` | 82 332 записи | Тестовая выборка |

**Типы атак в датасете (9 классов + Normal):**
- Fuzzers (~41%), Exploits (~34%), Generic (~13%), DoS (~6%), Reconnaissance (~3%), Backdoors (~1.7%), Shellcode (~1%), Analysis (~0.15%), Worms (~0.04%).

Для задачи бинарной классификации все типы атак объединены в один класс `label=1`.

### 3.3 Предобработка и EDA

**Основные шаги предобработки:**
1. Загрузка CSV-файлов и проверка структуры (45 колонок, пропусков нет).
2. Кодирование категориальных признаков (`proto`, `service`, `state`) через `LabelEncoder` из `scikit-learn`.
3. Сохранение энкодеров в `artifacts/label_encoders.pkl` для использования в продакшне.
4. Разделение на признаки (X) и целевую переменную (y).

**Ключевые наблюдения из EDA** (ноутбук `notebooks/01_eda_wifi_traffic.ipynb`):
- Классы относительно сбалансированы (44% атак в train, 55% в test).
- Аномальные потоки имеют значительно большие значения `sbytes`, `rate`, `sload` и меньшую длительность `dur`.
- Сильная корреляция с `label` у признаков `ct_srv_src`, `ct_state_ttl`, `sload`, `dload`.
- Выбросы (метод IQR) преимущественно в классе Attack, что подтверждает их природу.

## 4. Модели и подходы

### 4.1 Базовые (baseline) модели

В ноутбуке `notebooks/02_feature_engineering_and_baseline.ipynb` реализованы три наивные стратегии:

| Модель | Описание | F1-Score | Комментарий |
|--------|----------|----------|-------------|
| **Always Normal** | Всегда предсказывает 0 | 0.000 | Показывает бесполезность угадывания мажоритарного класса |
| **Threshold by mean bytes** | Порог по среднему `sbytes + dbytes` | 0.099 | Слишком грубый порог |
| **Rules by quantiles** | Классификация по квантилям | 0.000 | Тривиальная задача, не релевантна детекции |

Baseline подтверждает, что задача нетривиальна и требует ML-моделей.

### 4.2 Улучшенные модели и эксперименты

В ноутбуке `notebooks/03_model_comparison.ipynb` обучены две модели:

**Isolation Forest (unsupervised):**
- 100 деревьев, `contamination=0.2`.
- Не использует метки при обучении — ищет статистические выбросы.
- Результат: F1 = 0.260, Recall = 0.179.
- **Недостаток:** пропускает 82% атак — неприемлемо для безопасности.

**LightGBM (supervised градиентный бустинг):**
- 100 деревьев, `learning_rate=0.1`, `max_depth=6`, `num_leaves=31`.
- Использует все 42 признака после LabelEncoding категориальных.
- Результат: **F1 = 0.895, Recall = 0.988, Precision = 0.818, Accuracy = 0.873**.

**Что улучшило метрики:**
- Кодирование категориальных признаков через LabelEncoder.
- Использование всех 42 признаков — LightGBM сам определяет важность.
- Разделение train/test по оригинальным файлам датасета (без случайного shuffle).

**Что не улучшило:**
- Попытки агрегации во временные окна (5-минутные) — дали всего 28 примеров в train, что критически мало. Отказались от этого подхода.

## 5. Экспериментальный протокол и результаты

### 5.1 Экспериментальный протокол

- **Разделение:** фиксированные train/test из оригинального датасета (175 341 / 82 332).
- **Seed:** `random_state=42` для воспроизводимости.
- **Метрики:** считались на test-выборке, не использованной при обучении.
- **Фиксация:** все эксперименты зафиксированы в ноутбуках.

### 5.2 Сравнение моделей по метрикам

| Модель / конфигурация | Описание | Accuracy | Precision | Recall | F1-Score | Комментарий |
|------------------------|----------|----------|-----------|--------|----------|-------------|
| Baseline (Always Normal) | Всегда предсказывает 0 | 0.449 | 0.000 | 0.000 | 0.000 | Стартовая точка |
| Baseline (Threshold) | Порог по среднему трафику | 0.414 | 0.321 | 0.058 | 0.099 | Слишком грубо |
| Isolation Forest | 100 деревьев, contamination=0.2 | 0.437 | 0.471 | 0.179 | 0.260 | Unsupervised, много пропусков |
| **LightGBM** | 100 деревьев, lr=0.1, depth=6 | **0.873** | **0.818** | **0.988** | **0.895** | **Финальная модель** |

### 5.3 Выбор финальной модели

В продакшн выбрана **LightGBM** по следующим причинам:

1. **Наилучший F1-Score (0.895)** — в 3.4 раза лучше Isolation Forest.
2. **Критически высокий Recall (0.988)** — модель пропускает лишь 1.2% атак.
3. **Приемлемый Precision (0.818)** — 18% ложных алертов, что допустимо для первого уровня фильтрации.
4. **Скорость инференса** — ~10 мс, что укладывается в требование real-time.
5. **Интерпретируемость** — встроенная важность признаков (топ-3: `ct_srv_src`, `ct_state_ttl`, `sload`).

**Trade-off:** LightGBM сложнее в развёртывании, чем пороговый baseline, и требует кодирования категориальных признаков. Однако выигрыш в качестве (F1 0.895 vs 0.099) многократно оправдывает эту сложность.

## 6. Архитектура решения и сервис

### 6.1 Архитектура пайплайна

Пайплайн состоит из четырёх логических слоёв:

1. **Клиент (Client)** — браузер, curl, SIEM-система.
2. **Docker-контейнер** — FastAPI-сервис с компонентами:
   - `Pydantic` — валидация входных данных.
   - `DataPreprocessor` — кодирование категориальных признаков.
   - `ModelPredictor` — инференс LightGBM.
   - `Response Formatter` — формирование JSON-ответа.
3. **Артефакты** — `lgbm_anomaly_model.pkl`, `label_encoders.pkl`.
4. **Конфигурация** — `configs/.env` с переменными окружения.

**Поток данных:** клиент → FastAPI (валидация) → Preprocessor (кодирование) → Predictor (инференс) → Response Formatter → клиент.

### 6.2 API и endpoints

Реализованы четыре эндпоинта:

| Эндпоинт | Метод | Описание |
|----------|-------|----------|
| `/` | GET | Информация об API |
| `/health` | GET | Проверка здоровья: статус, загруженность модели, timestamp |
| `/predict` | POST | Принимает JSON с 42 признаками, возвращает предсказание |
| `/docs` | GET | Интерактивная документация Swagger UI |

**Формат запроса `/predict`:**
```bash
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{"dur": 0.000011, "proto": "udp", "service": "-", "state": "INT",
       "spkts": 2, "dpkts": 0, "sbytes": 496, "dbytes": 0,
       "rate": 90909.0902, "sttl": 254, "dttl": 0, ...}'
```

**Формат ответа:**
```bash
{
  "prediction": 0,
  "label": "Normal",
  "confidence": 0.9523,
  "attack_probability": 0.0477,
  "processing_time_ms": 12.34
}
```

## 7. Наблюдаемость, конфигурация и безопасность

Кратко опишите:

1. **Логи и наблюдаемость:**
   - что логируется: 
      - События запуска и остановки сервиса (lifespan-события FastAPI);
      - Статус загрузки модели и предобработчика (успех или ошибка с указанием пути к файлу);
      - Каждый запрос к /predict: предсказанная метка (label), уверенность модели (confidence) и время обработки в миллисекундах;
      - Ошибки валидации входных данных Pydantic (уровень WARNING);
      - Критические ошибки инференса модели (уровень ERROR).
   - как можно понять, что сервис работает корректно:
      - Эндпоинт /health: возвращает JSON со статусом сервиса и флагами model_loaded: true и preprocessor_loaded: true. Используется для liveness/readiness проверок;
      - Логи: наличие сообщений об успешной загрузке модели и отсутствие записей уровня ERROR;
      - Метрики времени: время обработки запроса (processing_time_ms) стабильно находится в диапазоне 10–50 мс, что соответствует требованиям real-time;

2. **Конфигурации:**
   - какие параметры вынесены в конфиги (`configs/`, `.env.example`):
      - MODEL_PATH — путь к файлу обученной модели (.pkl);
      - ENCODERS_PATH — путь к файлу энкодеров категориальных признаков;
      - API_HOST и API_PORT — хост и порт для запуска ASGI-сервера (uvicorn);
      - LOG_LEVEL — уровень детализации логов (DEBUG, INFO, WARNING, ERROR).
   - что можно перенастроить без изменения кода: Для изменения путей к артефактам (например, при обновлении модели), смены порта при деплое на другой сервер или включения режима отладки (DEBUG) достаточно отредактировать файл configs/.env и перезапустить контейнер или сервис. Изменять исходный код (src/) не требуется.

3. **Безопасность:**
   - как вы избегали коммита секретов (токенов, паролей): 
      - В репозитории хранится исключительно шаблон configs/.env.example без реальных значений;
      - Файл с реальными переменными configs/.env добавлен в .gitignore и физически не может быть закоммичен;
      - Настроены правила .gitignore для исключения любых файлов с расширениями .env, .env.local.
   - нет ли в репозитории “чувствительных” данных:
      - Логи: в логах намеренно не сохраняются IP-адреса клиентов и тела HTTP-запросов, чтобы исключить утечку потенциально чувствительных данных. Логируются только метаданные и результаты инференса;
      - Датасет: используемый датасет UNSW-NB15 является синтетическим и не содержит персональной информации (PII). Все IP-адреса и идентификаторы сгенерированы искусственно;
      - Артефакты: бинарные файлы моделей (.pkl) и большие CSV-файлы с сырыми данными также исключены из Git через .gitignore, в репозитории хранятся только их описания и визуализации.

## 8. Ограничения и дальнейшая работа

Будьте честны:

- какие у проекта есть ограничения (качество, скорость, устойчивость, объём данных и т.п.):
   - Синтетический характер датасета. UNSW-NB15 сгенерирован в лабораторных условиях инструментом IXIA PerfectStorm. Распределение реального сетевого трафика в корпоративной или домашней сети может существенно отличаться от лабораторного, что снижает обобщающую способность модели при деплое в продакшн;
   - Отсутствие временных меток. В датасете нет реальных timestamp'ов, поэтому агрегация во временные окна выполнена искусственно. Сервис работает на уровне отдельных сетевых потоков (flows), а не временных рядов, что ограничивает возможность выявления длительных атак;
   - Несбалансированность типов атак. Редкие классы представлены крайне мало: Worms (~0,04%), Analysis (~0,15%), Shellcode (~1%). Модель плохо детектирует эти типы атак, хотя в бинарной постановке (норма/атака) это не критично;
   - Нет мониторинга дрейфа данных. Модель не отслеживает изменение распределения входных признаков в продакшне, что может привести к деградации качества без видимых признаков.
   - Только бинарная классификация. Модель определяет «норма/атака», но не классифицирует конкретный тип атаки (DoS, Exploits, Fuzzers и т.д.), хотя в датасете есть признак attack_cat с 9 классами. Это ограничивает возможности для расследования инцидентов;
   - Нет онлайн-обучения. Модель не обновляется на новых данных без полного переобучения и пересборки Docker-образа.
   - Нет версионирования моделей. Модели хранятся как .pkl-файлы без использования MLflow или DVC, что затрудняет откат к предыдущей версии.
- что бы вы сделали, если бы было больше времени:
  - улучшения модели: Многоклассовая классификация. Обучить модель для предсказания конкретного типа атаки (DoS, Exploits, Fuzzers, Reconnaissance, Backdoors, Shellcode, Worms, Analysis, Generic) вместо бинарной метки. Это позволит SOC-аналитикам быстрее реагировать на инциденты.  Ансамбли моделей. Комбинировать LightGBM с XGBoost, CatBoost и нейросетевыми моделями (например, LSTM для временных рядов) через stacking или voting для повышения точности и устойчивости. Обнаружение дрейфа данных. Интегрировать библиотеки типа Evidently AI или NannyML для мониторинга распределения входных признаков и автоматических алертов при обнаружении drift.
  - улучшения сервиса: Авторизация и rate limiting. Добавить JWT-токены или API-ключи для аутентификации клиентов, а также ограничение частоты запросов (например, 100 запросов/минуту на ключ) через middleware FastAPI. Мониторинг. Интегрировать Prometheus для сбора метрик (количество запросов, latency, ошибки) и Grafana для визуализации дашбордов. Добавить алерты в Slack/Telegram при аномалиях. Кэширование. Использовать Redis для кэширования частых запросов с одинаковыми входными данными, что снизит нагрузку на модель.
  - улучшения данных и экспериментов: Аугментация данных. Применить SMOTE или ADASYN для генерации синтетических примеров редких классов атак (Worms, Analysis), что улучшит Recall на этих классах. Гиперпараметрическая оптимизация. Применить Optuna или Hyperopt для автоматического поиска оптимальных гиперпараметров LightGBM (learning_rate, max_depth, num_leaves, min_child_samples). Интерпретируемость. Интегрировать SHAP (SHapley Additive exPlanations) для объяснения каждого предсказания: какие признаки внесли наибольший вклад в решение модели. Это повысит доверие к модели у SOC-аналитиков.
